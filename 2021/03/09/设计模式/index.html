设计模式总结

本文最后更新于：1 个月前

设计模式

设计模式是java这类面向对象程序设计语言的一大基础，也是很多企业在面试经常考察的重点。

总结了六大原则和三大类，以及一些常用的设计模式。

六大原则

  1. 开闭原则：对类内封闭，对类外扩展。不修改原有代码而添加新的代码实现功能修改和添加。
  2. 里氏代换原则：任何基类出现的地方，都可以用子类来实现，保证了继承和复用。
  3. 依赖倒转原则：依赖抽象实现而不是具体实现。
  4. 接口隔离原则：多个接口之间保持隔离，降低耦合
  5. 迪米特法则（最少知道原则）：各个实体应当尽量少的与其它实体发生相互作用，是使得相互独立。
  6. 合成复用原则：尽量使用合成/聚合的方式，而不是继承。

三大类

  • 行为型：包括策略模式、模板方法模式、观察者模式等
  • 创建型：包括工厂模式、抽象工厂模式、单例模式等
  • 结构型：包括适配器模式、装饰器模式、组合模式，代理模式等等

单例模式

使用场景：

但面对一个类只能有一个实例的情况，比如点击显示窗口只能有一个窗口出现，再点击显示窗口只是弹出之前显示的窗口而已，而不是每次点击都会出现一个新的窗口。

设计思想：

通过私有化类的构造函数使得其它类无法实例化该类，并在类内提供一个唯一的实例化对象，并提供一个公有方法传递给类外。

具体代码：

public class Singleton {
    //1.实例化必须为私有静态变量
    private static Singleton instance;
    //2.构造方法必须为私有，只能通过类内的方法访问
    private Singleton(){};
    //3.通过公共静态的方法向外界实例化类
    public static Singleton getInstance(){
        //被实例化之后将不再继续实例化，直接返回之前实例化好的对象
        if(instance == null){
            instance = new Singleton();
        }
        return instance;
    }
    /*这里需要考虑的一点是：
       对于多线程的话需要加锁，防止当两个线程同时调用getInstance方法时都会实例化，从而导致产生了两个实例化对象*/
}

/*多线程实现，其中加锁在条件判断之后提升了性能，减少了线程加锁的次数
public class Singleton {
    //1.实例化必须为私有静态变量
    private static Singleton instance;
    private static readonly Object syncRoot = new Object();
    //2.构造方法必须为私有，只能通过类内的方法访问
    private Singleton(){};
    //3.通过公共静态的方法向外界实例化类
    public static Singleton getInstance(){
        //被实例化之后将不再继续实例化，直接返回之前实例化好的对象
        if(instance == null){
            lock(syncRoot){ instance = new Singleton();
            }
        }
    }
    return instance;
}*/

那么在客户端代码，将不再以new的形式来生成对象：

public class Main {
    public static void main(String[] args){
        //关键部分，如果按照下面的方式实例化将报错
        //Singleton s1 = new Singleton();
        //Singleton s2 = new Singleton();
        Singleton s1 = Singleton.getInstance();
        Singleton s2 = Singleton.getInstance();
        if(s1.equals(s2)){
            System.out.println("true");
        }else{
            System.out.println("false");
        }
    }
}
//输出true，表示两个实例化对象其实是同一个

工厂方法模式

使用场景：

工厂方法模式的使用主要用于当类的内部不希望被类外所知道，通过一个工厂类来实现实例化，外部不需要关注类的内部，只需要通过工厂实现实例化就即可。

设计思想：

对于一个不希望被客户端所了解的类，只需要通过第三者-工厂类来实现即可，客户端告知工厂类需要的对象的类型和相关属性，工厂类“生产”出对应的对象给客户端。从而降低了类间的耦合。

具体代码：

以实现一个四则运算为例子，客户端给工厂运算符，而工厂负责实现具有该运算符运算能力的实例。

//operation.java

public class Operation {
    private double _numberA = 0;
    private double _numberB = 0;

    public void setNumberA(double value){
        this._numberA=value;
    }
    public void setNumberB(double value){
        this._numberB=value;
    }
    public double getNumberA(){
        return this._numberA;
    }
    public double getNumberB(){
        return this._numberB;
    }
    public double getResult(){
        double result=0;
        return result;
    }
}
//operationAdd.java

public class OperationAdd extends Operation{
    @Override
    public double getResult(){
        double result=this.getNumberA()+this.getNumberB();
        return result;
    }
}
//operationSub

public class OperationSub extends Operation{
    @Override
    public double getResult(){
        double result=this.getNumberA()+this.getNumberB();
        return result;
    }
}
//operationDiv

public class OperationDiv extends Operation{
    @Override
    public double getResult(){
        double result=0;
        double _numberB=this.getNumberB();
        try{
            if(_numberB!=0) result=this.getNumberA()/_numberB;
        }catch (Exception e){
            System.out.println("被除数不能为0"+e);
            throw e;
        }

        System.out.println("被除数为0!");
        return result;
    }
}
//工厂类
public class OperationFactory {
    public static Operation createOperator(String operate){
        Operation ope=null;
        switch(operate){
            case "+":
                ope=new OperationAdd();
                break;

            case "-":
                ope=new OperationSub();
                break;

            case "/":
                ope=new OperationDiv();
                break;
        }
        return ope;
    }
}
//客户端代码
public class main {
    public static void main(String[] args){
        OperationFactory opeF = new OperationFactory();
        Operation ope = opeF.createOperator("/");
        ope.setNumberA(2);
        ope.setNumberB(0);
        System.out.println(ope.getResult());
    }
}

可以看到，客户端对operationAdd、operationSub和operationDiv等类的访问是通过工厂类实现的。因此，可以说，这三个类对客户端是透明的。对于实际而言，也是如此，我们并不需要关注物品生产的方法，我们只需要知道我们要的是什么类型的物品，其他的交给生产者（工厂类）去实现就好了。

抽象工厂模式

使用场景：

由工厂模式，客户端可以不需理会类的内部设计，但是，如果需求发生改变，我们就需要修改工厂类，在工厂类中添加不同的实例化，这就违背了开闭原则（对类内封闭修改）。此时该怎么办呢，这里就可以考虑依赖倒转原则，将工厂类也依赖于抽象类而不是具体类。这样，便可以根据不同的需要设计不同的工厂类。

设计思想：

建立一个工厂类的抽象父类，其它工厂类继承于它。

策略模式

使用场景：

前面有说到工厂模式是通过创建不同的对象给予客户端。而策略模式也差不多，策略模式是在context类中根据客户端的需求选择不同的实现算法。工厂模式和策略模式两者的差别可以以一个场景进行理解：

你去餐厅点餐，服务员给你菜单，让你选择你所需要的菜品，而你不需要关注这些菜的具体烹饪方法。你点完单之后，服务员过了一会把菜都端了上来。这就是工厂模式。而策略模式则不同。同样的，你去餐厅点餐，此时，服务器给你的菜单不是一条条列出来的菜名了，而是各种烹饪方法，比如红烧的，加多少盐，加多少糖等等。你选择一种烹饪方法（算法），服务员在后台根据你选择的烹饪方法给你上菜。总而言之，工厂模式，客户端要的是对象，策略模式，客户端更多关注算法层面。

设计思想：

建立一个context类，客户端通过给context类参数，以此选择不同的算法。

模板方法模式

使用场景：

当多个对象之间具有太多重复代码时，可以提取一个模板类，所有子类在此模板类上加以进一步实现，提高代码的复用性，可维护性。

设计思想：

提取公共代码，提高复用性。主要利用多态思想，通过父类声明子类引用。

具体代码：

//Template.java
public class Template {
    String answer1,answer2;
    public Template(){
    }
    public void problem1(){
        System.out.println("这是问题1");
    }
    public void problem2(){
        System.out.println("这是问题2");
    }
    public void answer1(){
        System.out.println("问题1的答案是："+this.answer1);
    }
    public void answer2(){
        System.out.println("问题2的答案是："+this.answer2);
    }
    public void setAnswer(String answer1,String answer2){
        this.answer1=answer1;
        this.answer2=answer2;
    }
}
//studentA.java
public class StudentA extends Template{
    public StudentA(String answer1,String answer2){
        super.setAnswer(answer1, answer2);
    }
}
//studetnB.java
public class StudentB extends Template{
    public StudentB(String answer1,String answer2){
        super.setAnswer(answer1, answer2);
    }
}
//客户端
public class Main {
    public static void main(String[] args){
        Template t;
        //父类声明子类引用
        t= new StudentA("a","b");
        t.problem1();
        t.problem2();
        t.answer1();
        t.answer2();
        //父类声明子类引用
        t= new StudentA("c","d");
        t.problem1();
        t.problem2();
        t.answer1();
        t.answer2();
    }
}

通过模板方法，整体代码的观赏性就大大提高了，子类StudentA和StudentB只需要给参数即可。

观察者模式

使用场景：

当一个类的变化会导致另外一个类相应发生变化时。通过观察者类来实现两者之间的变化通知。

设计思想：

引入“第三者”类负责变化通知。

具体代码：

//subject.java
//抽象父类
public abstract class Subject {
    public abstract void Attach(Observer o);
    public abstract void Detach(Observer o);
    public abstract void Notify();
}
//抽象观察者类
public abstract class Observer {
    public abstract  void  Update();
}
//具体的子类
import java.util.ArrayList;

public class ConcreteSubject extends Subject{
    //被观察者集合
    ArrayList<Observer> obsers;
    public ConcreteSubject(){
        obsers=new ArrayList<Observer>();
    }
    public void Attach(Observer o){
        obsers.add(o);
    };
    public void Detach(Observer o){
        obsers.remove(o);
    };
    public void Notify(){
        //通知被观察者集合中的所有人
        for (Observer i:obsers
             ) {
            i.Update();
        }
    };
}
//具体的观察者子类
public class ConcreteObserver extends Observer{
    String name;
    public ConcreteObserver(String name){
        this.name = name;
    }
    public  void  Update(){
        System.out.println(this.name+"被subject提醒更新了");
    };
}
//客户端代码
public class Main {
    public static void main(String[] args){
        ConcreteObserver o1 = new ConcreteObserver("li");
        ConcreteObserver o2 = new ConcreteObserver("zhao");
        ConcreteSubject  s = new ConcreteSubject();
        s.Attach(o1);
        s.Attach(o2);
        s.Notify();
    }
}

组合模式

使用场景：

部分与整体不需要分离考虑，或者说，部分可以看成整体，整体也可以看成部分的情况。例如：省有财务管理，市也有财务管理，县也有财务管理，那么对于财务管理的逻辑中（不考虑数据库是否一致问题），他们是可以被当成相同属性对待的，即使省本身包含了市，市本身包含了县。我们需要统一对待部分与整体的关系时（树结构的分支和叶子节点）可以考虑组合模式。

设计思想：

通过继承的思想，所有的部分和整体都继承统一的父类，那么我们只需要将其看成父类对象即可。

具体代码：

//组件父类
public abstract class Component {
    protected String name=null;
    public abstract void add(Component c);
    public abstract void remove(Component c);
    public abstract void display(int depth);
}
//组合模式中的叶子节点类，部分类
public class Leaf extends Component{
    private String name;
    public Leaf(String name){
        this.name = name;
    }
    //透明方式，为了实现leaf和composite对于外界来说是透明的，没有任何区别，但其实没有任何用处
    //安全方式的话，在Component抽象类中不要声明这两个方法，而在Composite单独书写为其特有的方法，
    //但是这样对于客户端就需要添加额外的代码来判断是composite还是Leaf。
    public void add(Component c){};
    public void remove(Component c){};
    @Override
    public void display(int depth){
        System.out.println(depth+this.name);
    };
}
import java.util.ArrayList;
import java.util.List;
//组合模式的整体类
public class Composite extends Component{
    private List<Component> L= new ArrayList<Component>();
    private String name=null;

    public Composite(String name){
        this.name = name;
    }
    public void add(Component c){
        L.add(c);
    };
    public void remove(Component c){
        L.remove(c);
    };
    @Override
    public void display(int depth){
        System.out.println(depth+this.name);
        depth+=2;
        for (Component c: L
             ) {
            //这个有个坑，如果直接在此输出会导致输出name都为空，需要进入子类中在子类的display方法中打印结果。
            //System.out.println(c.name);
            //这里打印的其实是c.component.name,而不是c.name,进入子类的display方法后再打印就是c.name了。
            c.display(depth);
        }
    };
}
//客户端代码
//最终以Composite实例化对象出发，遍历所有部分和整体的实例化对象而不需要判断
public class Main {
    public static void main(String[] args){
        Composite root = new Composite("根节点");
        Composite branch1 = new Composite("分支节点");
        Leaf l1 = new Leaf("叶子一");
        Leaf l2 = new Leaf("叶子二");
        root.add(l1);
        root.add(l2);
        root.add(branch1);

        Leaf l3 = new Leaf("叶子三");
        Leaf l4 = new Leaf("叶子四");
        branch1.add(l3);
        branch1.add(l4);
        //root.remove(l2);
        //root.remove(branch1);
        root.display(0);
    }
}

适配器模式

使用场景：

这种模式其实本质上有点亡养补牢的感觉：对于一个外来的类或者说前期设计有缺陷的类，针对我们的接口无法实现，那么我们将利用第三者“适配器”来改善这个问题，使得这个类可以和我们提供的接口匹配。

设计思想：

引入adapter类来填补接口不匹配问题

具体代码：

//这里以NBA球员和外来需要翻译的球员为例
public abstract class Player {
    protected String name;
    //这里出现一个基础知识点，在定义了有参构造函数之后，无参的默认构造函数就失效了，
    //那么对于继承该类的子类而言，就必须显式的调用该有参构造函数或者在父类显式的申明无参默认的构造函数。
    public Player(){};
    public Player(String name){
        this.name = name;
    }
    public abstract void attack();
    public abstract void defense();
}
//中心
public class Center extends Player{
    public Center(String name){
        super(name);
    }

    public void attack(){
        System.out.println(this.name+"攻击");
    };
    public void defense(){
        System.out.println(this.name+"防守");
    };
}
//后卫
public class Guards extends Player{
    public Guards(String name){
        super(name);
    }

    public void attack(){
        System.out.println(this.name+"攻击");
    };
    public void defense(){
        System.out.println(this.name+"防守");
    };
}
//外来球员
public class ForeignForwards{
    //私有化属性来表示与其它两种center和guards不同之处
    private String name;
    public ForeignForwards(String name){
        this.name = name;
    }

    public void attack(){
        System.out.println(this.name+"攻击");
    };
    public void defense(){
        System.out.println(this.name+"防守");
    };
}
//通过翻译实现外来球员和其它球员无法匹配问题，适配器类
public class Translator extends Player{
    private ForeignForwards f = new ForeignForwards("姚明");
    public Translator(){};
    public void attack(){
        f.attack();
    };
    public void defense(){
        f.defense();
    };
}
//客户端代码
public class Main {
    public static void main(String[] args){
        Player a = new Center("11");
        Player b = new Guards("22");
        Player c = new Translator();
        a.attack();
        a.defense();
        b.attack();
        b.defense();
        c.attack();
        c.defense();
    }
}

装饰器模式

使用场景：

对一个类进行修改添加功能时，但是为了保证开闭原则，我们不能修改源码再编译部署（开闭原则的对内封闭）。我们只能通过添加类（开闭原则的对外扩展）。通过装饰器可以对原来的类添加属性和功能。

设计思想：

通过重写父类的方法，除了添加自己的属性和功能外，最后调用父类的该方法。其次需要在装饰器类中设置特定的方法来连接所有实例并执行最外层的实例的重写方法。

具体代码：

//基类
public class Person {
    public Person(){}
    private String name;
    public Person (String name){
        this.name=name;
    }
    public void show(){
        System.out.println("装饰的"+name);
    }
}
//装饰器类
public class Finery extends Person{
    protected Person p;
    //关键代码，连接所有实例
    public void setPerson(Person p){
        this.p=p;
    }
    @Override
    public void show(){
        p.show();
    }
}
public class FineryA extends Finery{

    String T;

    @Override
    public void show(){
        T = "skirt";
        super.show();
        System.out.println("修饰属性");
    }
}
public class FineryB extends Finery{
    @Override
    public void show(){
        this.addFunction();
        super.show();
    }
    public void addFunction(){
        System.out.println("添加的函数功能");
    }
}
//客户端代码
public class Main {
    public static void main(String[] args){
        Person p = new Person();
        FineryA a = new FineryA();
        FineryB b = new FineryB();
        a.setPerson(p);
        b.setPerson(a);
        b.show();
    }
}

代理模式

使用场景：

代理模式与之前的方法也类似，通过第三个类“代理类”建立客户端和具体类之间的关系，而导致客户端只需要和代理沟通即可，而无需关注具体类。

设计思想：

代理类实例化具体类，并执行具体类的方法。而客户端只考虑代理类。降低了类间耦合。

具体代码：

public abstract class Subject {
    public abstract void  require();
}


public class ConcretePerson extends Subject{
    String name;
    public ConcretePerson(String value){
        this.name = value;
    }
    public void function1(){
        System.out.println("代理实现");
    }
    @Override
    public void require(){
        System.out.println(this.name+"要求");
    }
}
//代理类
public class Proxy extends Subject{
    ConcretePerson p = null;

    public Proxy(){
        if(p==null){
            p = new ConcretePerson("LI");
        }
    }
    public void function1(){
        this.p.function1();
    }
    @Override
    public void require(){
        this.p.require();
    }
}
//客户端代码
public class Main {
    public static void main(String[] args){
        Proxy pro = new Proxy();
        pro.function1();
        pro.require();
    }
}

总结

以上都是较为常用的设计模式，但不排除其它设计模式一定不会被使用，还是应该看具体场景。总结来说，最为关键的原则就是开闭原则，依赖倒转。基本都是围绕着几大中心思想来实现的：

  1. 保证每个类只有单一职责
  2. 保证高内聚，低耦合。类内聚合，但类间尽可能解耦。
  3. 引用第三类来降低两个主要类之间的耦合
  4. 通过继承和多态来实现扩展便于之后的代码修改。
  5. 提取公共代码。
